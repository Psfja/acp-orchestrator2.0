Section 2/4 — ACP 通信适配层

核心挑战：不同 Agent 的 ACP 启动方式不同，需要统一适配

为什么需要适配层

虽然都实现了 ACP 协议，但每个 Agent 的启动命令、参数格式、实现细节略有差异：

各 Agent 的 ACP 启动方式
Agent 启动命令 类型 备注

适配层设计 — Adapter Pattern

ACPAdapter 接口 + 具体实现
# ─── 抽象接口 ───

class ACPAdapter(ABC):
    """所有 ACP Agent 的统一适配接口"""

    @abstractmethod
    def get_launch_command(self) -> list[str]:
        """返回启动命令，如 ['claude-code-acp']"""
        pass

    @abstractmethod
    async def spawn(self) -> tuple[asyncio.StreamWriter, asyncio.StreamReader]:
        """启动子进程，返回 stdin/stdout 管道"""
        pass

    @abstractmethod
    async def initialize(self, writer, reader, session_config: dict) -> str:
        """
        ACP 握手:
        1. 发送 initialize request (协议版本协商)
        2. 发送 session/new (注入 CWD + system_prompt)
        3. 返回 session_id
        """
        pass

    @abstractmethod
    async def send_prompt(self, writer, session_id, prompt: str) -> None:
        """发送 session/prompt — 把任务交给 Agent"""
        pass

    @abstractmethod
    async def read_updates(self, reader) -> AsyncIterator[dict]:
        """流式读取 session/update 通知"""
        pass

    @abstractmethod
    async def close_session(self, writer, session_id) -> None:
        """关闭会话"""
        pass


# ─── 具体实现示例 ───

class ClaudeCodeAdapter(ACPAdapter):
    def get_launch_command(self):
        return ["claude-code-acp"]

class CodexAdapter(ACPAdapter):
    def get_launch_command(self):
        return ["npx", "@zed-industries/codex-acp"]

class GeminiAdapter(ACPAdapter):
    def get_launch_command(self):
        return ["gemini", "--acp"]

ACP 通信流程 (标准 JSON-RPC 2.0)

一次完整的 ACP 会话
# Step 1: 启动子进程
$ claude-code-acp
# 子进程启动，等待 stdin 输入 JSON-RPC 消息...

# Step 2: initialize (协议握手)
→ {"jsonrpc":"2.0","id":1,"method":"initialize","params":{
      "protocolVersion":"0.11.0",
      "clientInfo":{"name":"acp-orchestrator","version":"0.1.0"},
      "capabilities":{"fs":{},"terminal":{}}
    }}
← {"jsonrpc":"2.0","id":1,"result":{
      "serverInfo":{"name":"Claude Code","version":"..."},
      "capabilities":{"fs":{},"terminal":{}}
    }}

# Step 3: session/new (创建会话 + 注入系统提示词)
→ {"jsonrpc":"2.0","id":2,"method":"session/new","params":{
      "cwd":"/path/to/project",
      "systemPrompt":"你是后端开发专家。你只负责编写后端API代码..."  ← 来自agents.yaml
    }}
← {"jsonrpc":"2.0","id":2,"result":{"sessionId":"sess-001"}}

# Step 4: session/prompt (发送任务)
→ {"jsonrpc":"2.0","id":3,"method":"session/prompt","params":{
      "sessionId":"sess-001",
      "prompt":[{"type":"text","text":"编写登录验证API，包含输入校验和错误处理"}]
    }}

# Step 5: 流式接收 session/update (Agent 工作过程)
← {"jsonrpc":"2.0","method":"session/update","params":{
      "sessionId":"sess-001",
      "update":{"type":"message","content":"我来编写登录API..."}
    }}
← {"jsonrpc":"2.0","method":"session/update","params":{
      "sessionId":"sess-001",
      "update":{"type":"message","content":"已完成，文件: auth.py"}
    }}

# Step 6: session/prompt 返回 (任务完成)
← {"jsonrpc":"2.0","id":3,"result":{
      "stopReason":"end_turn",
      "message":{"type":"text","text":"已完成登录验证API的编写..."}
    }}

# Step 7: 关闭会话
→ {"jsonrpc":"2.0","id":4,"method":"session/close","params":{"sessionId":"sess-001"}}

关键设计点

🔌 统一适配接口

每个 Agent 类型一个 Adapter 类

新加 Agent 只需实现 ACPAdapter 接口

在 agents.yaml 中指定 adapter 类型

💉 系统提示词注入

在 session/new 阶段注入 systemPrompt

提示词来自 agents.yaml 配置

限制 Agent 只做自己专长的事

🔄 流式输出转发

session/update 的通知实时被 Logger 捕获

Agent 的工作过程对用户完全可见

不等待完成才输出 — 边做边看
